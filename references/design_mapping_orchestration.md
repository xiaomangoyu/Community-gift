# 设计字段映射与编排

这份文档描述主播资料进入生图工作流时，哪些字段应该确定性保留，哪些字段交给模型综合判断。

## 核心原则

当前阶段，CSV 只提供最少的人类设计信号，不再提供完整设计参数。

更高优先级原则：主播数据只做差异化，不负责定义产品骨架，也不负责提供 native prompt。

### 2026-05-16 补充共识：CSV 只保留名字、颜色、元素和氛围

当前 `host_design_params_simplified.csv` 只应读取：

- 主播名字
- 主色 / 辅助色
- 代表符号
- 直播氛围

其他信息在这个阶段都不放进 CSV，也不由主流程读取，包括：

- 推荐输出类型
- 主体形态
- 主文字 / 副文字
- 材质语言
- 装饰强度
- 必须避免
- 映射理由
- prompt 术语或英文生图短语

这些不是没用，而是应该交给后续编排层、效果库和全局产品合同兜底。CSV 的职责是提供干净的主播差异化信号，不能变成半条 prompt。

### 2026-05-18 补充共识：颜色分为主色和配色

颜色不再只作为一串 prompt 词直接透传。当前分两层：

- `主色`：主播/主题的识别锚点，用来决定外壳、核心、包边、手柄或底部节点的主落位。
- `配色`：由本地颜色库扩展出的 3-5 个产品安全色，用来保证黑底可抠图、材质层次和整体稳定性。

颜色库位置：`references/color_palettes.json`。

CSV 里的主色和辅助色仍然重要，但它们只作为选择颜色库和确定主色锚点的输入；最终 prompt 使用颜色库输出的 `main_color` 和 `palette`。例如 CSV 写黑色时，产品主色会安全转译为烟灰/枪灰/银灰，而不是生成纯黑棒身。

### 2026-05-18 补充共识：中文 prompt 保留，只降说明书感

当前生图接的是 Seedream，中文 prompt 本身不是问题，不需要为了英文而英文。

短期继续沿用参考 prompt 的长中文结构；优化目标不是重写模板，而是轻量降低“说明书 / 产品图解”味道：

- 正向 prompt 保持中文审美 brief 和产品合同。
- 不大幅改变参考 prompt 的段落顺序和硬约束。
- 容易诱导画面生成说明板、标注、图解的否定清单尽量放在 negative prompt。
- 如后续接 LLM smoothing，只允许润色语言，不允许改主符号、主色锚点、配色、材质、名字位置和构图合同。

## 2026-05-16 新共识：好 Prompt 是主干，主播数据是 Delta

当前失败的关键不是字段不够，而是字段进入 prompt 后破坏了好 prompt 原本稳定的结构、节奏和写法。

后续工作流必须反过来：

```text
好 prompt / 好图
  -> 结构化成 prompt 模板和审美规则
  -> deterministic template 按槽位生成最终 prompt
  -> 主播数据只作为少量变量填入
```

而不是：

```text
主播字段
  -> 拼接成一条越来越长的 prompt
  -> 试图让模型同时理解产品、审美、符号、文字和禁用项
```

无论主播数据给了多少信息，最终 prompt 的结构和写法都应该接近已经验证过的好 prompt。宁愿忽略一部分主播数据，也不能破坏好 prompt 的骨架。

### 2026-05-16 补充共识：名字、元素、配色必须落到棒体上

template-first 不是“无主播信息的通用棒”。在保持好 prompt 骨架的前提下，每条最终 prompt 必须完成三个落位：

1. 主播名字必须出现在应援棒上方中间，作为灯头上沿 / 上方中央的一体化发光主铭牌；不是画面外标题，不放在手柄上，不做产品说明卡文字。
2. 主播元素至少有一个要明确呼应到应援棒本体，优先成为灯头外轮廓、边缘浮雕、晶体镶嵌、底座压纹或局部发光节点。
3. 主播配色至少有一组要明确呼应到应援棒本体，落在外壳、中央核心、包边、手柄或底部节点上；不能只存在于背景或抽象气质描述里。

如果主播资料太多，仍然只保留少量 delta，但这三个落位不能被省略：

```text
主播名字上方中间铭牌
1 个主播主元素进入棒体结构
1 组主播主色 / 辅色进入棒体材质与高光
```

### 好 Prompt 结构化目标

用户后续会提供若干条已经验证过的好 prompt。它们不应该被原样复制，而应该拆成结构化规则：

| 层 | 需要提炼的内容 |
| --- | --- |
| 产品摄影合同 | 画幅、背景、视角、主体占比、完整性、边缘留白 |
| 产品骨架 | 灯头、中央核心 / 铭牌、连接件、手柄、底盖 |
| 视觉句式 | prompt 的段落顺序、描述节奏、关键词密度、重复强调的位置 |
| 材质公式 | 树脂、珐琅、晶体、珠光、金属边的组合方式 |
| 复杂度公式 | 允许几个主元素，哪些元素只能做边缘/浮雕/小节点 |
| 负面空间 | 哪些内容绝对不能出现，例如白卡片、分屏、近景、平面贴纸 |
| 可变槽位 | 主播名上方铭牌、主符号、辅助符号、主色、辅助色、材质偏好、禁用项 |

最终生成 prompt 时，默认不再让 LLM 写终稿；模型只提供结构化槽位，脚本负责保持好 prompt 的结构和语气，只替换少数变量。

### 主播数据的降级原则

主播数据有冲突时，按下面优先级处理：

```text
好 prompt 结构 > 产品骨架 > 可抠图与完整性 > 主符号 > 颜色/材质 > 辅助符号 > 口号/副文字/零碎备注
```

如果主播字段太多，只保留：

- 主播名字，固定进入上方中间主铭牌
- 1 个主符号
- 1 组主色 / 辅助色
- 直播氛围

其他符号、口号、工具、小梗、次级 emoji、材质、输出类型、禁用项、运营备注和 prompt 术语全部忽略或交给编排兜底。

### Pass / Not Pass 暂由人工定义

VLM 自动审核节点当前默认关闭。原因是它现在无法代表真实审美判断，还会把自动重试带偏。后续即使接回，也只作为失败类型分类和批量复盘辅助，不能直接把评估意见拼回最终 prompt。

当前阶段的流程是：

```text
生成候选图
记录 prompt / raw response / 图片
人工判断是否基本符合标准
用户确认标准后，再定义 pass / not pass 规则
```

后续只有在人工标准稳定后，才把 VLM 重新接回验收节点。

## 当前代码执行策略

默认走 `template-first`：

```text
CSV
  -> 抽取少量槽位：主播名字、颜色、代表符号、直播氛围
  -> 套稳定参考 prompt 骨架
  -> Seedream 4.5 并发出图
```

默认不再调用 LLM 重新设计整条 prompt，不再让 `notes`、`mapping_reason`、`secondary_text` 等长信息支配画面。

旧的 LLM/planner 编排保留为显式 legacy 模式：

```bash
python -m community_gift.cli --csv examples/sample.csv --legacy-llm-design
```

在新的好 prompt 库结构化完成前，legacy 模式只用于对照，不作为主工作流。

先稳定一个通用产品骨架：

```text
黑底
单支完整应援棒
非纯黑可抠图棒身
清楚灯头、连接件、手柄、底盖
中央空白发光铭牌 / 核心
主播名字在上方中间主铭牌
至少一个主播元素呼应到棒体
至少一组主播配色呼应到棒体
2-3 个大体块装饰
无白卡片、无分屏、无近景
```

主播字段只允许影响：

- 上方中间主铭牌文字
- 一个主元素和最多一个辅助元素
- 主色 / 辅助色
- 氛围气质

主播字段不应该改变产品摄影合同、完整性、背景、镜头距离、手柄结构或文字策略。否则每条数据都会把模型拉去重新发明一款物体，批量结果就会变得无序。

工作流应该分三层：

1. 数据层：只保留主播名、颜色、符号和氛围。
2. 编排层：把结构化信息压成 `DesignConcept`、`TextRenderPlan`、`PromptPlan`，先做取舍，再进 prompt。
3. 生成层：由 plan-driven reference prompt 锁定黑底、完整单支、实体厚度、发光核心、手柄和底盖等硬约束。

## 字段边界

### 当前阶段确定读取

当前 `host_design_params_simplified.csv` 只承载低污染输入：

| 来源字段 | 目标字段 | 用途 |
| --- | --- | --- |
| 主播名字 | `host_name` | 唯一身份文本，不在 CSV 阶段改写 |
| 主色 | `primary_color` | 主视觉颜色 |
| 辅助色 | `secondary_color` | 边缘、高光、节点色 |
| 代表符号 | `symbols` | 1-2 个自然语言元素 |
| 直播氛围 | `live_vibe` | 只影响气质，不直接变成画面清单 |

### 当前阶段不读取

这些字段先不放进简化 CSV，也不让主流程读取：

- 主体形态
- 主文字 / 副文字
- 材质语言
- 装饰强度
- 必须避免
- 推荐输出类型
- 设计置信度
- 映射理由
- 任何 prompt 术语或英文生图短语

## 编排建议

### 0. 先生成主概念

不要直接从 CSV 进入生图 prompt。每条主播数据先由编排层根据名字、颜色、元素和氛围生成一个可审阅的主概念：

| 字段 | 作用 |
| --- | --- |
| `main_proposition` | 一句话说明这支应援棒的主设计命题 |
| `primary_symbol` | 唯一主符号，优先塑造灯头主轮廓 |
| `retained_elements` | 最多 2-3 个保留元素 |
| `discarded_elements` | 明确丢弃的符号、口号、配件、贴片 |
| `silhouette` | 灯头主轮廓，例如 winged / crown / rounded mascot / brim-inspired |
| `complexity_level` | 低或中，控制 prompt 里允许的装饰量 |

这一步的目标是把“字段清单”变成“设计命题”。例如：

```text
羽翼 + 闪电晶芯 + 甜酷电光
-> 羽翼包住闪电晶芯；只保留一个主轮廓和一个小型发光细节。
```

### 1. 先筛测试样本

优先顺序：

1. 主播名字明确
2. 主色 / 辅助色明确
3. 代表符号明确
4. 直播氛围能区分气质
5. 样本之间元素差异明显

对照样本可以保留在更完整的资料表里，但简化 CSV 当前只服务本轮小样测试。

### 2. 再组装设计上下文

给模型的输入应该只来自低污染字段：

- 主播名字：`host_name`
- 色彩：`primary_color` / `secondary_color`
- 物理符号：`symbols`
- 氛围：`live_vibe`

不要让模型从 `notes`、`mapping_reason`、`body_form`、`material_language_hint` 或任何 prompt 术语里猜设计。

当前代码会把这些字段进一步整理进：

- `design_concept`：由元素和氛围生成主概念。
- `text_plan`：当前只基于主播名字制定文字策略。
- `prompt_plan`：摄影合同、产品合同、材质、禁用项由全局模板兜底。

### 3. 最后套 reference 质量约束

reference prompt 应该锁定的是产品质量，而不是统一形状：

- `1:1` 正方形画幅，主工作流默认 `2048x2048`；只有快速小样才临时用 `1024x1024`
- 单支完整产品
- 黑色背景
- 灯头、连接件、手柄、底盖完整
- 中央实心发光核心
- 文字嵌入或浮雕
- 圆润树脂、珐琅、果冻晶体、克制金属
- 符号长在轮廓上，不是平贴表面

主体形态不再来自 CSV 字段，而由编排层根据 `代表符号` 和 `直播氛围` 推导。例如羽翼、土豆、皇冠、蝴蝶、帽檐都只是元素信号，不是直接 prompt。

## 文字策略

文字是硬约束，不是普通装饰。主播名 / 主文字不能破形、不能拼错、不能被模型改写。

当前编排先采用保守策略：

- 图片里只允许一个保真文字目标：优先 `host_name` 的可读文本版，`primary_text` / `community_name` 只作为备选和设计语义。
- `secondary_text`、主播签名、口号、小字带都不交给图像模型自由生成。
- 短拉丁文字使用 `one_shot_text`：唯一文字必须放在灯头最大的中央发光核心或中央嵌入铭牌里，保持水平、居中、清晰。
- 长名字、多语言、复杂标点使用 `post_overlay_text`：图像模型只生成空白中央铭牌 / 发光核心，不生成任何文字或伪字母。
- 手柄必须保持无字结构，只允许材质、纹理、握持造型和无字装饰；任何手柄文字、竖排字、斜排字、伪字母都判为失败。
- one-shot 模式下中央核心 / 铭牌不能留空；post-overlay 模式下中央核心 / 铭牌必须留作干净文字承载区。
- 当前不依赖 VLM 返回 `text_fail`。人工复看时只要出现乱码、错拼、额外签名、随机小字、文字斜贴或位置跑偏，下一轮就切到两阶段思路：先生成空白铭牌产品图，再用程序化文字层覆盖精确文本。

对于多语言、emoji、超长主播名，优先考虑后置文字层；不要要求图像模型直接画完整复杂名字。
