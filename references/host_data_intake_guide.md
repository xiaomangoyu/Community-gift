# 主播数据整理指南

这份文档用于把研发给的大量原始数据，整理成应援棒生图真正需要的主播设计信息。

## 当前结论

研发给的数据通常会有很多和设计无关的字段，例如：

- 内部 ID
- 国家/地区编码
- 业务状态
- 时间戳
- 审核状态
- 运营标签
- 粉丝数、播放量、流水等指标
- 技术字段
- 空字段或重复字段

这些字段不应该直接进入 prompt。

我们要做的是从原始数据里提取“能影响视觉设计”的少量信息。

## 最终需要的标准字段

当前项目读取的 CSV 字段是：

```csv
主播名字,主播图片,社群名字,内容类型,直播氛围,主播性格,主应援色,辅助色,代表符号,禁用元素,备注
```

每个字段的作用：

| 字段 | 是否重要 | 用途 |
| --- | --- | --- |
| 主播名字 | 必须 | 当前阶段唯一身份文本 |
| 主应援色 / 主色 | 必须 | 主视觉颜色 |
| 辅助色 | 可选 | 边框、高光、节点色 |
| 代表符号 | 必须 | 1-2 个自然语言元素，不写 prompt 术语 |
| 直播氛围 | 必须 | 影响气质，例如甜酷、高能、清爽、温柔 |

当前简化 CSV 不读取主播图片、社群名字、内容类型、主播性格、禁用元素、备注、主体形态、材质语言、装饰强度或输出类型。这些信息可以留在原始资料表里，等编排层需要时再使用。

## 原始字段筛选原则

### 保留

保留能回答这些问题的字段：

- 这个主播叫什么？
- 这个主播有什么固定颜色？
- 这个主播有什么代表 emoji / 符号？
- 这个主播的直播氛围是什么？

### 合并

多个字段表达同一件事时，合并到一个标准字段：

| 原始字段例子 | 合并到 |
| --- | --- |
| nickname, display_name, anchor_name, creator_name | 主播名字 |
| avatar, cover, profile_image, image_url | 主播图片 |
| fanclub_name, group_name, community, clan | 社群名字 |
| category, vertical, content_tag, live_type | 内容类型 |
| vibe, persona, style_tag, streamer_tags | 直播氛围 / 主播性格 |
| primary_color, theme_color, fan_color | 主应援色 |
| emojis, symbols, props, interests | 代表符号 |
| forbidden, banned, avoid, risk_note | 禁用元素 |

### 丢弃

默认丢弃：

- 纯技术 ID
- 数据库主键
- 创建/更新时间
- 审核流状态
- 运营任务状态
- 空字段
- 完全重复字段
- 和视觉无关的数值指标

注意：粉丝数、排名、流水这类字段通常不直接进 prompt。除非我们明确要把它转译成“顶流、满分、高能、庆典”等视觉气质。

## 推荐工作流

### 第一步：从 Lark / 飞书导出原始数据

如果 Lark CLI 可用，目标是导出成 CSV 或 JSON。

如果 CLI 不可用，直接在飞书表格里导出 CSV 也可以。

建议先只导出 20-50 行样本，不要一开始全量。

保存到：

```text
data/raw/
```

例如：

```text
data/raw/hosts_from_rd.csv
```

### 第二步：先看表头，不看全量

先只看字段名，判断哪些可能有用。

目标是做一张字段映射表：

```csv
原始字段,标准字段,处理方式,备注
nickname,主播名字,保留,
avatar_url,主播图片,保留,
category,内容类型,保留,
created_at,,丢弃,技术字段
fans_count,备注,可选,只用于判断顶流感，不直接进prompt
```

### 第三步：抽样 5-10 行人工判断

不要一开始全自动。

先抽 5-10 个主播，人工看：

- 映射出来的信息是否像人话
- 有没有把无用技术字段塞进去
- 有没有漏掉关键设计信息
- 主播之间是否有足够差异

### 第四步：生成标准输入 CSV

整理成项目可读格式：

```text
data/processed/hosts_design_input.csv
```

字段必须是：

```csv
主播名字,主播图片,社群名字,内容类型,直播氛围,主播性格,主应援色,辅助色,代表符号,禁用元素,备注
```

### 第五步：先跑 dry-run

先只生成结构化设计，不急着出图。

```bash
python -m community_gift.cli \
  --csv data/processed/hosts_design_input.csv \
  --effect-library references/effects.json \
  --dry-run \
  --max-rows 5
```

检查输出：

```text
outputs/structured_designs.csv
outputs/structured_designs.json
```

重点看：

- 主体形态是否有多样性
- 颜色是否合理
- 代表符号有没有被转译成产品结构
- 是否过度依赖爱心
- 是否塞入无关数据

### 第六步：确认映射后再出图

只挑 3-5 条差异明显的主播先出图。

不要一次全量生成。

## 判断字段有没有用的标准

一个字段是否有用，看它能不能影响下面任一项：

- 灯头主体形态
- 主色/辅助色
- 中央文字
- 装饰符号
- 材质气质
- 装饰复杂度
- 禁用约束

如果不能影响这些，就不要进入标准 CSV。

## 低价值但可间接使用的数据

有些研发字段本身不适合进 prompt，但可以转译成备注：

| 数据 | 可转译为 |
| --- | --- |
| 高粉丝数 / 高排名 | 顶流感、舞台中心感 |
| 高互动 | 高能、热闹、庆典感 |
| 新主播 | 清新、成长、初舞台感 |
| 多地区语言 | 多语言文字、异域装饰、地区色彩 |
| 游戏类标签 | 电竞、机械、能量、速度感 |
| 美妆/时尚标签 | 珠光、镜面、花朵、皇冠、轻奢 |
